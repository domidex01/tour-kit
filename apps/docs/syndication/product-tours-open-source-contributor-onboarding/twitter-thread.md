## Thread (6 tweets)

**1/** Homebrew has millions of users but only tens of maintainers. That drop-off ratio is typical for open-source projects. The onboarding between "using the code" and "improving the code" is broken.

**2/** The data is clear on what works:
- README file → +55% productivity
- CONTRIBUTING.md → +17%
- Good First Issues → +13% more contributors
- Mentorship → +46% productivity

But less than 1% of projects use interactive walkthroughs. That's the gap.

**3/** Traditional onboarding is passive: dump a CONTRIBUTING.md and hope people read it. Product tours flip this — walk contributors through the codebase, dev setup, and PR process step by step.

4-6 steps. 20 minutes to implement. Persists progress across sessions.

**4/** The AI slop problem makes this urgent. 36M new GitHub devs in 2025, many using AI to generate PRs. Teaching quality standards *before* the first submission reduces low-quality PRs and saves reviewer time.

**5/** The really interesting part: retention. Mike McQuaid says retention matters more than attraction. GitHub's 2026 guidance: build "durable systems" not individual mentorship.

Progressive nudges (first PR → reviewer → maintainer) automate the contributor ladder.
