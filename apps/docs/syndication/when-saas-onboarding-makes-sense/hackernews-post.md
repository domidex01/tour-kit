## Title: When SaaS onboarding tools make more sense than code libraries (from a library author)

## URL: https://usertourkit.com/blog/when-saas-onboarding-makes-sense

## Comment to post immediately after:

I build Tour Kit, an open-source React library for product tours. Thought I'd write an honest piece about when our competition (SaaS tools like Appcues, Userpilot, Pendo) is actually the right choice.

The short version: it depends on who owns onboarding. If a PM iterates on flows weekly and can't write React, a visual editor removes the developer bottleneck entirely. If engineering owns it, a library gives you full control.

Some data points from the piece:
- Retool's 2026 report: 35% of enterprises have replaced SaaS with custom builds, but 78% still plan to build more
- Appcues estimates $70,784 in year-one costs for a single in-house onboarding flow
- SaaS pricing ranges from $89/mo to $595/mo, scaling with MAU count
- At 100K+ MAU, SaaS costs can exceed $50K/year vs. a one-time library integration

The article identifies six specific scenarios where SaaS wins (non-technical ownership, speed, bundled analytics, cross-channel, compliance, scale) and the counterarguments where libraries are clearly better (cost at scale, performance, design consistency, vendor lock-in).

Curious to hear from folks who've made this decision recently — which way did you go and why?
