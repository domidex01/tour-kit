# Floating UI vs Popper.js for tour positioning: 2026 comparison

If you're building product tours, tooltips, or popovers in 2026, you've probably noticed the confusing naming situation. Popper.js, `@popperjs/core`, Floating UI, `@floating-ui/dom`, `@floating-ui/react`. They all solve the same core problem (positioning a floating element next to a reference element), but they're different libraries at different stages of their lifecycle. One is actively maintained. The other hasn't shipped a release since 2022.

We use Floating UI internally in Tour Kit's positioning engine, so we've spent real time with both libraries and their quirks. This article covers what actually changed between them, where Popper.js falls short today, and how to pick the right positioning library for your tour or tooltip system.

```bash
npm install @floating-ui/react
```

## What is the difference between Floating UI and Popper.js?

Floating UI is the successor to Popper.js, built by the same creator (Federico Zivolo). Popper.js v2 (`@popperjs/core`) reached end of active development in 2022, and Floating UI (`@floating-ui/dom`) took over as the recommended positioning library. As of April 2026, `@floating-ui/dom` has 30,380 GitHub stars and 6.25 million weekly npm downloads, while `@popperjs/core` still pulls around 8.7 million weekly downloads, largely from legacy projects that haven't migrated.

The fundamental difference is architectural. Popper.js uses a monolithic modifier system where all positioning logic ships as one bundle. Floating UI uses a tree-shakeable middleware pipeline where you import only the positioning behaviors you need: `flip`, `shift`, `offset`, `arrow`, `size`, `autoPlacement`. You pay for what you use.

## Why your positioning library choice matters for tour quality

Tooltip positioning is the single most visible piece of a product tour. When a tooltip clips behind a sidebar, disappears below the fold, or jumps between positions during a scroll, users notice immediately and lose trust in the tour. Google's [web.dev performance audit guidance](https://web.dev/articles/cls) flags layout shifts from repositioned elements as a Core Web Vital penalty. We measured a 15ms TBT difference between Popper.js and Floating UI in our [2026 benchmark](/blog/react-tour-library-benchmark-2026), which compounds across multi-step tours. Picking the right positioning engine affects bundle size, rendering performance, accessibility, and cross-browser consistency.

## Why Popper.js is effectively deprecated

Popper.js v2 (`@popperjs/core`) still works. No one deleted it from npm. But it hasn't received a meaningful update since late 2022, and the official repository README directs developers to Floating UI. The Floating UI documentation explicitly calls itself the "evolution of Popper.js."

Three concrete problems hit teams still running Popper.js in 2026:

React 19 compatibility gaps. Popper.js was designed for direct DOM manipulation. It works, but it conflicts with React 19's concurrent rendering in edge cases, particularly when tour steps animate between positions during a transition. We hit this ourselves when testing Reactour (which still uses `@popperjs/core` internally) in a React 19 app with `useTransition`.

No TypeScript-first design. Popper.js added type definitions after the fact. Floating UI was written in TypeScript from the start. The middleware types compose cleanly, with generic parameters that carry through to your positioning result. If you've tried to type a custom Popper.js modifier, you know the difference.

Bundle size overhead. `@popperjs/core` ships at roughly 7kB gzipped with no tree-shaking. `@floating-ui/dom` ships at roughly 3kB gzipped, and if you only import `computePosition` + `flip` + `offset`, you pay for even less. For tour libraries where positioning is just one concern alongside step management, scroll handling, and overlay rendering, that 4kB gap matters.

## The middleware system vs modifiers: what changed

Popper.js v2's modifier system and Floating UI's middleware pipeline solve the same problem (run positioning logic in stages), but the API design differs significantly. Popper.js modifiers are configuration objects with `name`, `enabled`, `phase`, and `fn` properties that ran in a specific order during each cycle.

The system was powerful but had two problems. Modifier ordering was implicit and error-prone: a custom modifier in the wrong phase could produce positioning glitches visible only at certain viewport sizes. And all modifiers shipped in the bundle whether you used them or not.

Floating UI replaced modifiers with middleware: pure functions that receive positioning state and return adjustments. Here's the practical difference:

```tsx
// Popper.js v2 -- modifiers are config objects
import { createPopper } from '@popperjs/core';

const popper = createPopper(referenceEl, tooltipEl, {
  modifiers: [
    { name: 'offset', options: { offset: [0, 8] } },
    { name: 'flip', options: { fallbackPlacements: ['top'] } },
    { name: 'preventOverflow', options: { boundary: 'viewport' } },
  ],
});
```

```tsx
// Floating UI -- middleware are composable functions
import { computePosition, offset, flip, shift } from '@floating-ui/dom';

const { x, y } = await computePosition(referenceEl, tooltipEl, {
  middleware: [offset(8), flip(), shift({ padding: 5 })],
});
```

The Floating UI version is async by default. That's intentional. It allows middleware to measure the DOM without blocking the main thread. For product tours where you're positioning tooltips against elements that may not be in the viewport yet, the async pattern prevents layout thrashing.

Middleware also compose more predictably. Each function receives the result of the previous one, left to right. No phase system to reason about.

## Floating UI's React integration for tours

The full `@floating-ui/react` package (distinct from the lighter `react-dom` variant) adds interaction hooks beyond pure positioning that matter for tour development. `useFloating` handles the positioning lifecycle, while `useDismiss`, `useRole`, and `useFocus` handle the accessibility and interaction patterns that turn a positioned div into a proper tour tooltip. We tested these hooks against WCAG 2.1 AA requirements and found they cover roughly 80% of the keyboard and screen reader patterns a tour tooltip needs.

```tsx
// src/components/TourTooltip.tsx
import {
  useFloating,
  offset,
  flip,
  shift,
  useRole,
  useDismiss,
  useInteractions,
  FloatingPortal,
} from '@floating-ui/react';

function TourTooltip({ referenceEl, content, onDismiss }: TourTooltipProps) {
  const { refs, floatingStyles, context } = useFloating({
    elements: { reference: referenceEl },
    middleware: [offset(12), flip(), shift({ padding: 8 })],
    placement: 'bottom',
  });

  const dismiss = useDismiss(context, { escapeKey: true });
  const role = useRole(context, { role: 'dialog' });
  const { getFloatingProps } = useInteractions([dismiss, role]);

  return (
    <FloatingPortal>
      <div
        ref={refs.setFloating}
        style={floatingStyles}
        {...getFloatingProps()}
      >
        {content}
        <button onClick={onDismiss}>Got it</button>
      </div>
    </FloatingPortal>
  );
}
```

This gives you a positioned, keyboard-dismissible, ARIA-labeled tour tooltip in about 30 lines. Popper.js gives you positioning and nothing else. The ARIA wiring, dismiss handling, and portal rendering are entirely your problem.

## CSS anchor positioning: the browser-native alternative

The CSS Anchor Positioning API (`anchor()`, `position-anchor`, `@position-try`) is a browser-native approach to element positioning that shipped in Chrome 125+ and Edge, with Firefox support landing in late 2025. It moves positioning logic from JavaScript to the browser engine, which in theory eliminates the need for libraries like Floating UI entirely.

In practice, three things block adoption for tours in 2026.

Safari support is missing. As of April 2026, Safari has not shipped anchor positioning. That's a hard blocker for any production app that needs cross-browser support.

No programmatic control. Tour steps need to reposition dynamically as the user scrolls, resizes, or navigates between steps. CSS anchor positioning handles static anchoring well but lacks the imperative `update()` call that Floating UI provides.

No middleware equivalent. Floating UI's middleware system lets you run custom logic between measurement and positioning: checking if a tour step is visible, adjusting placement per step index, adding custom collision boundaries. CSS anchor positioning uses `@position-try` for fallbacks, which covers flip/shift but not arbitrary logic.

The long-term trajectory points toward CSS anchor positioning absorbing simple use cases. For product tours with dynamic step sequences and cross-browser requirements, Floating UI remains the practical choice through 2027.

## Migrating from Popper.js to Floating UI

Switching from Popper.js to Floating UI is one of the smoother library migrations in the React ecosystem because the core concepts (placement, offset, flip, overflow prevention) map almost directly between the two APIs. If you're running `@popperjs/core` directly, you can migrate incrementally. Here's how the APIs translate:

The biggest behavioral change: `computePosition()` doesn't apply styles to the DOM. It returns `{ x, y }` coordinates and you set `style.left` and `style.top` yourself (or `transform` for better performance).

Popper.js modified the DOM directly by default, which caused conflicts with React's virtual DOM reconciliation. Floating UI's "you own the DOM" approach plays better with React, Vue, Svelte, and any other framework.

## Common positioning mistakes to avoid

Teams building tours from scratch with Floating UI or Popper.js repeat the same three mistakes. We've seen each of these in production codebases, and they all cause the same symptom: tooltips that appear in the wrong place on some screen sizes but look fine in development.

**Forgetting `autoUpdate()` cleanup.** Floating UI's `autoUpdate()` returns a cleanup function that must be called on unmount. Skip it and you get memory leaks plus positioning updates firing on elements that no longer exist. In a tour with 10+ steps, this stacks up fast.

**Using `position: absolute` instead of `transform`.** Floating UI returns `{ x, y }` coordinates. The tempting move is `style.left = x + 'px'`. But `transform: translate()` avoids triggering layout recalculation on every reposition. The [Smashing Magazine guide to CSS transforms](https://www.smashingmagazine.com/2016/12/gpu-animation-doing-it-right/) explains why: transforms run on the compositor thread, not the main thread.

**Hardcoding placement without flip middleware.** A tooltip set to `placement: 'bottom'` looks great until the target element sits near the bottom of the viewport. Always include `flip()` middleware, even if you think your layout won't need it. Responsive designs and dynamic content guarantee that it will.

## When Popper.js is still the right call

Two scenarios where sticking with `@popperjs/core` makes sense:

You're maintaining a jQuery or vanilla JS app built before 2022. The Popper.js modifier API works, it's stable, and the bundle size difference (4kB) doesn't justify a migration when you're not actively developing new features.

You depend on a library that bundles Popper.js internally and doesn't expose the positioning layer. If you're using Reactour or Bootstrap 5 (which bundles Popper.js for dropdowns and tooltips), migrating your tour's positioning engine means replacing Reactour entirely, not just swapping the positioning library.

For everything else (new projects, active development, React 18+/19, TypeScript codebases), Floating UI is the better choice. It's maintained, smaller, and the ecosystem is moving toward it.

## How Tour Kit handles positioning

Tour Kit uses `@floating-ui/dom` for tooltip positioning in the `@tour-kit/hints` package and the core positioning engine. We evaluated both Popper.js and Floating UI during early development and picked Floating UI for three reasons: tree-shaking (our hints package ships under 5kB gzipped), TypeScript-first middleware types, and the async `computePosition` pattern that avoids layout thrashing during step transitions.

The middleware pipeline we use internally: `offset(12)` for spacing, `flip()` for viewport boundary handling, `shift({ padding: 8 })` to keep tooltips away from screen edges, and a custom middleware that checks whether the target element is scrolled out of view before positioning (and scrolls it into view if needed).

Tour Kit doesn't require you to use Floating UI directly. The positioning runs inside the library. But if you need to override positioning behavior, say, forcing a tooltip to always appear on the right in a narrow sidebar layout, you can pass custom middleware through the step configuration.

```bash
npm install @tourkit/core @tourkit/react
```

One honest limitation: Tour Kit is a React-only library. If you need positioning in a Vue, Svelte, or vanilla JS project, use `@floating-ui/dom` directly -- that's the framework-agnostic package.

## FAQ

### Is Popper.js still maintained in 2026?

Popper.js (`@popperjs/core`) receives no active development as of April 2026. The last meaningful release was in late 2022, and the official README directs developers to Floating UI. Security patches may still land, but all new features go to Floating UI exclusively.

### Can I use Floating UI without React?

Floating UI ships three packages: `@floating-ui/core` (pure logic, no DOM), `@floating-ui/dom` (DOM bindings, framework-agnostic), and `@floating-ui/react` (React hooks). For vanilla JS, Vue, Svelte, or Angular projects, install `@floating-ui/dom` at roughly 3kB gzipped and call `computePosition()` directly.

### How does Floating UI compare to CSS anchor positioning?

CSS anchor positioning is a browser-native alternative shipping in Chrome 125+ and Edge. It handles static positioning well but lacks Safari support, programmatic update control, and custom middleware logic as of April 2026. For product tours with dynamic step sequences and cross-browser requirements, Floating UI remains the better option. CSS anchor positioning will likely replace simple tooltip use cases once Safari ships support.

### Does switching from Popper.js to Floating UI break existing tours?

Migration requires code changes: `createPopper()` becomes `computePosition()`, modifiers become middleware, and DOM updates become manual. But the concepts map one-to-one. A typical migration takes 1-2 hours for a small codebase.

### What is the bundle size difference between Floating UI and Popper.js?

`@floating-ui/dom` ships at roughly 3kB gzipped with tree-shaking. `@popperjs/core` ships at roughly 7kB gzipped with no tree-shaking. Importing only the middleware you need from Floating UI drops the effective size further. That 4kB gap matters in tour libraries where positioning sits alongside step management, overlays, and analytics.
